What the Helm!?

Tags:
  • helm
  • ci/cd
August 24, 2019

Or, “things I with I knew about Helm before using it as the continuous delivery solution for my Kubernetes clusters!"

I’ve been running Helm as a Continuous Delivery tool for over one year. At QuintoAndar we’ve used it to perform over 300 releases daily - I’m certain there are far larger use cases out there, but that is probably large-is enough to get to see the good, the bad and the ugly side of most tools.

Now, don’t get me wrong, Helm is certainly not bad. But it can be complex, and it doesn’t always behave in obvious ways. So I’ve decided to write about this, about the pitfalls that won’t be clear to you if you’ve just read the documentation, but that may become painfully so when Helm is managing hundreds of workloads in production for you.

Who is this piece aimed at?

  • People using Helm as their main as their main Continuous Delivery tool
  • People thinking about using Helm as their main Continuous Delivery tool

That’s it. Also, this is not an introductory article! I will skip over the basic concepts, so if you are new to Helm please read the QuickStart Guide first. After that, by all means, feel free to come back and enjoy the ride.

Picking the Helm Version

Let’s start with the easy part!

Please, please grab the latest stable Helm 2 version. If you must have something tested by me, run version 2.13, this will give you the --atomic flag, pretty handy for continuous delivery. If that is still not enough, consider at least version 2.12. That’s the one I’ve used for the longest time and it includes a few bugfixes that I consider essential (at least one of those bugs can leave your release irrecoverable).

Also, just so there’s no confusion: as of august 2019, Helm 3 is still in beta, not ready for production, and will change a lot of things. I won’t cover it at all.

Initializing Helm

Now, there’s a couple of things you’ll want to get right when setting up Helm in a production environment. Some of them are covered just fine in the QuickStart Guide, some of them are not, and some are covered in ways I completely disagree with. So pay attention!

First of all, set the --history-max parameter. This controls how many previous instances of each release is kept. The documentation suggests a value of 200 which is of course downright insane! Since each Helm Release translates into a Kubernetes ConfigMap or Secret, the number of resources can grow very fast and put a real strain in your cluster. It may even break Helm itself. In practical terms a Release simply gives you the ability to use the command helm rollback to restore to a previous state. That is certainly useful, but you don’t need to store 200 releases. Keeping 20, 10 or even 5 is just fine.

If you already have Tiller running without a history limit, it is possible to reconfigure this, but you will need to upgrade Tiller or at least edit its deployment. See this issue for more information.

Another concern is, of course, security. Actually, scratch that, no one cares about security! So feel free to skip to the next section.

For the two of you who are tagging along, I recommend this great write-up from Bitnami Engineering, Exploring the Security of Helm. There are a lot of different strategies and they greatly depend on your situation, but I think the article’s conclusion sums it up nicely:

If you have no 3rd-party services that talk to tiller: Restart tiller with –listen=localhost:44134 flag.

If you have 3rd-party services that talk to tiller: Configure TLS authentication.

If you need to offer different levels of access via tiller: Complicated. Multiple tiller instances, or Helm-crd.

The key takeaway is that you probably do not need to bother with TLS, you can just limit Tiller’s GRPC webserver access to “localhost” instead. This way it’s only possible to connect to it through Kubernetes API server port-forward capabilities, conveniently used by the Helm client. If that does not work for you, use TLS. Then the best source is the QuickStart documentation, under TLS/SSL for Helm and Tiller.

Also, remember that I said that each Helm release is a ConfigMap or a Secret? By default, it’s a ConfigMap, which is a bit unsafe. If, for example, you were to use Helm to deploy a Secret, Helm would - rather unhelpfully - store its contents in a ConfigMap. Still, you can override that behavior, just don’t forget to do so!

Finally, unless you are running some sort of Medieval-era Kubernetes, you should use RBAC. The choices here are pretty much either giving Tiller the “cluster-admin” role, or restricting it to one or several namespaces.

For this example, I’m creating a tiller role with permissions to manage resources in the whole cluster:

$ kubectl -n kube-system create serviceaccount tiller && \
  kubectl create clusterrolebinding tiller \
          --clusterrole cluster-admin \
          --serviceaccount=kube-system:tiller

This gives us an initialization command that is more or less like this:

$ helm init \          # helm init command
    --history-max=5 \  # short helm history size
    --override 'spec.template.spec.containers[0].command'='{/tiller,--storage=secret}' \  # override storage for secret instead of configmap
    --override 'spec.template.spec.containers[0].args'='{--listen=localhost:44134}' \     # block all outside connections, only port-forward allowed
    --tiller-image gcr.io/kubernetes-helm/tiller:v2.14.3 \  # use latest release
    --service-account=tiller  # use service-account we just created

Alright! Now that Helm is running, we can start breaking things.

Helm is the Captain Now! Not You!

I’m going to run some Helm commands as example, to illustrate my points. These are real tests against a local Minikube cluster.

Here I’m using a handy helper command. This creates a sample chart that consists of a Nginx deployments.

$ helm create mychart

Now, to deploy my lonely Nginx server with Helm:

$ helm install ./mychart --name nginx

This has created a Helm release in our Kubernetes cluster. Remember that I’m assuming that you, the reader, is already familiar with the Helm basics! Now let’s assume that someone else, maybe a uninformed developer, used kubectl (or the Kubernetes dashboard, does not really matter) to scale up the number of replicas of our Nginx server.

$ kubectl scale deployments nginx-mychart --replicas=3
deployment.extensions/nginx-mychart scaled

Maybe this could have been a service that was under too much stress and lacked autoscaling. Well, the reasons don’t matter. For our hypothetical example, let’s assume we try to scale it back to 1 instance, but now using Helm. If you read the sample chart documentation, or use helm inspect mychart command, you will quick see this is the replicaCount parameter:

$ helm upgrade nginx ./mychart --set replicaCount=3 --wait

So now we are back to one server, right? Well, no.

$ kubectl get po     
NAME                             READY   STATUS    RESTARTS   AGE
nginx-mychart-56d94dd744-pd4mp   1/1     Running   0          3m1s
nginx-mychart-56d94dd744-pxc6z   1/1     Running   0          3m1s
nginx-mychart-56d94dd744-vncqg   1/1     Running   0          9m9s

What? Still three pods? Yet, helm history shows the latest release as DEPLOYED. Also, helm get shows the expected value of replicaCount: 1 everywhere. So Helm tells me I have one server, yet Kubernetes clearly tells me I have three! What madness is this?

Could it be reset-values? Or maybe is it reuse-values? Believe me, you do not want to go down that path.

Alright, what is happening here?

Nothing is happening, after all, it is a Feature, not a Bug.

Helm expects that you’re not modifying manifests outside of the Helm system.

You see, as a rule of thumb, at every new release, Helm compares all the incoming data with the latest successful release. Everything else gets discarded. Any modification to a resource that Helm itself didn’t make is ignored. In the example above, from Helm’s point of view, Nginx first release had one replica. The next release, the one we’re trying to make, also has one replica. The fact that at some point someone used kubectl to manually scale to 3 replicas is irrelevant! Thus, Helm does nothing and says nothing.

Still, if you know about it, you can be prepared: once you start using Helm as the means of deployment into your cluster, make sure every single change is made by Helm. And if you make any change outside Helm, try to undo it eventually. Otherwise it might sit there forever.

Helm is the Captain now. Either he is in charge, or no one is.

The Upgrade Command

So, as I hinted before, helm upgrade has a lot of Flags. A lot. And this causes all sorts of problems for a continuous delivery tool, because every potential combination is a new code path that has to be tested for bugs. It’s a new set of use cases that it has to support. When I combine --cleanup-on-fail with and without --wait, will it behave any differently? (yeah, it does).

So, which ones should we be using? Well, let me tell you first which ones you should NOT be using.

The First Time is the Worst

Now, you probably do not want to be using the install command in a continuous delivery pipeline. The reason is that it only works for the first release! For every successive release, you need the upgrade command. But what if you want something that works on both? Like, you know, kubectl apply? Oh, of course!

$ helm apply
Error: unknown command "apply" for "helm"
Run 'helm --help' for usage.

Hah, don’t be silly! Did you actually think things would start to make sense now? In fact, you use helm upgrade --install. Why did this win over helm install --upgrade is beyond me.

Continuous Delivery

We know this happens: most of us install something and once it works forget about it forever. Upgrading is risky and scary (despite what these 3 month “long” term support disclosures say) .

So this has been a long rant. I didn’t set out to convince anyone to start or stop using Helm, just meant to talk about some considerations that might interest others meaning to use it for continuous delivery. For my part, I do find Helm Templates quite powerful, and Helm Charts quite practical, but at the same time I find it lacking as a release manager. A colleague has looked into GitOps and we’re testing ArgoCD and so far the results are very promising.
